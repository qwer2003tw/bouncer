# Sprint 34-003: Trust Session IP binding upgrade

> GitHub Issue: #sec-004
> Priority: P0
> TCS: 8
> Generated: 2026-03-12

---

## Problem Statement

Trust Session IP binding 目前只有 **informational** 等級：當 IP mismatch 時只記錄 warning log + emit metric，**不會 block** 操作。

### 現狀

在 `src/trust.py` `should_trust_approve()` ~line 411：

```python
if stored_ip != current_ip:
    logger.warning(f"IP mismatch: stored {stored_ip}, current {current_ip}")
    emit_metric('trust_session_ip_mismatch', 1)
    # 但不 return False — 繼續執行
```

**問題**：
1. IP binding 沒有強制執行 → 無法防禦 session hijacking
2. 不同來源 IP 是正常情況：Telegram callbacks (91.108.x.x) vs MCP local calls (127.0.0.1)
3. 簡單加 `return False` 會 break 正常流程

### 設計意圖

Telegram callbacks 和 MCP calls 有不同的 source IP：
- **Telegram callback**: 來自 Telegram servers (91.108.x.x)
- **MCP call**: 來自本機 (127.0.0.1 或 container IP)

因此 **不能硬 block**，需要 configurable 策略。

### 目標

新增 `ip_binding_mode` 配置，支援 3 種模式：
1. **`strict`**: IP mismatch → block（回傳 False）
2. **`warn`** (default): IP mismatch → log + metric，不 block
3. **`disabled`**: 不檢查 IP

## Root Cause

1. 原始設計時考慮到 Telegram/MCP IP 不同，選擇 informational only
2. 缺少 configurable policy → 無法根據不同環境/安全需求調整
3. `tests/test_trust_ip_binding.py` 已存在，但缺少 strict mode 測試

## User Stories

**US-1: Strict mode for high-security environments**
As a **security-conscious Bouncer admin**,
I want to enable strict IP binding mode,
So that session tokens cannot be used from different IPs (防禦 session hijacking).

**US-2: Warn mode for mixed IP scenarios**
As a **Bouncer operator using both Telegram and MCP**,
I want IP mismatch to log warnings but not block operations,
So that both Telegram callbacks and MCP calls work without friction.

**US-3: Disabled mode for development**
As a **Bouncer developer**,
I want to disable IP binding checks entirely,
So that I can test freely without IP restrictions.

## Scope

### In Scope

1. 新增 `ip_binding_mode` 配置（environment variable + config.py）
2. 修改 `should_trust_approve()` 支援 3 種模式
3. 新增 strict mode 測試（`tests/test_trust_ip_binding.py`）
4. 文件說明 3 種模式的使用場景

### Mode Behavior

| Mode | IP Match | IP Mismatch | Use Case |
|------|----------|-------------|----------|
| `strict` | ✅ approve | ❌ block | High-security prod, single IP source |
| `warn` (default) | ✅ approve | ⚠️ log + approve | Mixed Telegram/MCP, need flexibility |
| `disabled` | ✅ approve | ✅ approve (no check) | Dev/test, debugging |

### Out of Scope

- 不改變 session creation 邏輯（IP 仍然儲存）
- 不新增 IP whitelist（簡化設計，用 mode 就夠）
- 不改變 Telegram callback source IP（無法控制）
- 不新增 per-project IP binding config（只支援全域設定）

## Acceptance Criteria

### AC-1: Strict mode blocks IP mismatch
```gherkin
Given ip_binding_mode is set to "strict"
And a trust session has stored_ip = "192.168.1.100"
When a request comes from current_ip = "192.168.1.200"
Then should_trust_approve() returns False
And a metric "trust_session_ip_blocked" is emitted
And a warning log is written
```

### AC-2: Warn mode allows IP mismatch
```gherkin
Given ip_binding_mode is set to "warn"
And a trust session has stored_ip = "192.168.1.100"
When a request comes from current_ip = "192.168.1.200"
Then should_trust_approve() returns True (不 block)
And a metric "trust_session_ip_mismatch" is emitted
And a warning log is written
```

### AC-3: Disabled mode skips IP check
```gherkin
Given ip_binding_mode is set to "disabled"
And a trust session exists (with any stored_ip)
When a request comes from any current_ip
Then should_trust_approve() returns True
And no IP check is performed
And no IP-related metrics or logs are emitted
```

### AC-4: Default mode is "warn"
```gherkin
Given ip_binding_mode is not explicitly set
When Bouncer starts
Then ip_binding_mode defaults to "warn"
```

### AC-5: Invalid mode falls back to "warn"
```gherkin
Given ip_binding_mode is set to "invalid_value"
When Bouncer starts
Then ip_binding_mode falls back to "warn"
And a warning log is written about invalid config
```

## Testing Strategy

1. **Unit tests** (`tests/test_trust_ip_binding.py`):
   - Strict mode: IP match → True, IP mismatch → False
   - Warn mode: IP match → True, IP mismatch → True (with log/metric)
   - Disabled mode: always True
   - Invalid mode → fallback to warn

2. **Integration tests**:
   - Strict mode: 實際 trust session + IP mismatch → 操作被拒絕
   - Warn mode: 實際 trust session + IP mismatch → 操作成功 + warning

3. **Regression tests**:
   - 現有 trust session tests 仍然 pass
   - 預設行為（warn mode）不改變

## Risk Analysis

| 風險 | 機率 | 影響 | 緩解 |
|------|------|------|------|
| Strict mode 誤 block 正常流量 | 中 | 高 | 預設 warn mode，文件明確說明使用場景 |
| Config 讀取失敗 → fallback 不正確 | 低 | 中 | fallback to warn（最安全的 default） |
| 測試覆蓋不足 → prod 發現 bug | 低 | 中 | 3 種 mode 都有完整測試 |
| Telegram callbacks 被 strict mode block | 中 | 高 | 文件說明：strict mode 只適合單一 IP source |

## Dependencies

- `src/trust.py` `should_trust_approve()` 函數（已存在）
- `tests/test_trust_ip_binding.py` 測試檔案（已存在）

## Configuration

### Environment Variable

```bash
# .env or AWS Lambda environment variables
BOUNCER_IP_BINDING_MODE=strict  # or "warn" or "disabled"
```

### Config.py

```python
# src/config.py
IP_BINDING_MODE = os.getenv('BOUNCER_IP_BINDING_MODE', 'warn').lower()

# Validate
if IP_BINDING_MODE not in ['strict', 'warn', 'disabled']:
    logger.warning(f"Invalid IP_BINDING_MODE: {IP_BINDING_MODE}, fallback to 'warn'")
    IP_BINDING_MODE = 'warn'
```

## Metrics

新增 CloudWatch metric：
- `trust_session_ip_blocked`: strict mode 下 IP mismatch 被 block 的次數
- 保留現有 `trust_session_ip_mismatch`: warn mode 下 IP mismatch 的次數
