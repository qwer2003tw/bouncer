# Sprint 34-001: Fix auto_approved notification bug

> GitHub Issue: #102, #116
> Priority: P0
> TCS: 3
> Generated: 2026-03-12

---

## Problem Statement

`auto_approved` 執行通知邏輯有 bug：當 auto-approved command 失敗時，Telegram 通知仍然顯示 ✅（成功），應該顯示 ❌（失敗）。

### 問題根因

在 `src/notifications.py` line ~500，判斷 auto-approved 命令是否失敗的邏輯使用 `result.startswith('❌')`：

```python
# notifications.py ~500
if result.startswith('❌'):
    # 失敗邏輯
else:
    # 成功邏輯
```

**問題**：
1. `result` 是完整的 command output + exit code wrapper，不一定以 `❌` 開頭
2. 存在偽陽性/偽陰性判斷 — 即使 exit code ≠ 0，也可能不以 `❌` 開頭
3. 同一檔案 line ~360 已有正確實作：`extract_exit_code(result)` 用於 trust session auto-approve path

### 目標

修正 `auto_approved` 通知邏輯，使用正確的 exit code 判斷方式（`extract_exit_code()`）替代 `startswith('❌')`。

## Root Cause

`src/notifications.py` 在 auto-approved 通知路徑使用了不可靠的字串前綴判斷，而不是已存在的 exit code 提取函數。

**Why did this happen?**
- 不同開發時間點實作的兩個路徑（trust session path 先實作 + 使用 `extract_exit_code`）
- 後續 auto-approved 路徑直接複製舊的 `startswith` 邏輯
- 缺少統一的 "判斷 command 成功/失敗" helper，導致邏輯分散

## User Stories

**US-1: Correct failure notification**
As a **Bouncer user**,
When an auto-approved command fails,
I want to see ❌ in the notification,
So that I know the command failed without checking logs.

**US-2: Consistent success/failure detection**
As a **Bouncer maintainer**,
I want all command result paths to use the same exit code detection logic,
So that we don't have inconsistent behavior across different approval paths.

## Scope

### In Scope

1. 修改 `src/notifications.py` ~line 500 的 auto-approved 通知邏輯
2. 將 `result.startswith('❌')` 替換為 `extract_exit_code(result)` 或 `_is_execute_failed(result)`
3. 新增 regression test：auto-approved command exit code 0 → ✅，exit code ≠ 0 → ❌
4. 驗證 `extract_exit_code()` 函數的正確性（已在 trust session path 使用）

### Out of Scope

- 不修改 command executor 的輸出格式
- 不統一所有 notification path（只修 auto-approved path）
- 不改變 trust session path（已經正確）
- 不新增 unified helper function（本 sprint 只修 bug，不重構）
