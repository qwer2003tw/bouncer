## Technical Context

### 現有架構
- `mcp_execute.py` `_submit_for_approval()`：建立 DDB approval request，TTL = `int(time.time()) + 300 + 60`（5分鐘 + buffer）
- `template.yaml`：已有 `SchedulerManageExpiry` IAM policy（`scheduler:CreateSchedule`, `scheduler:DeleteSchedule`, `scheduler:GetSchedule`）
- `src/scheduler_service.py`：已存在 EventBridge Scheduler 封裝

### 方案選擇
| 方案 | 成本 | 複雜度 | 說明 |
|------|------|--------|------|
| EventBridge Scheduler | 極低（前14M次/月免費）| 中 | 精確控制提醒時間，與現有 SchedulerManageExpiry IAM 對齊 |
| DynamoDB Streams + Lambda | $0.02/100萬讀取 | 高 | 需額外 Streams trigger，過度設計 |
| Step Functions wait | 只適合 deploy flow | N/A | 不適用 execute flow |

**選擇：EventBridge Scheduler**（成本最低，架構最乾淨）

## Implementation Phases

### Phase 1: Scheduler Setup（template.yaml）
- 確認 `SchedulerManageExpiry` IAM policy 涵蓋 `scheduler:DeleteSchedule`（TTL 到期後清理）
- 新增 `ExpiryNotifierLambda`（或複用 NotifierLambda）處理 scheduler 觸發事件

### Phase 2: Core Logic（mcp_execute.py / scheduler_service.py）
- `_submit_for_approval()` 建立 approval request 後，同時建立 EventBridge Schedule
  - 觸發時間：`TTL - 60`（過期前 60 秒）
  - Target：NotifierLambda / ExpiryNotifierFunction
  - Payload：`{"action": "expiry_warning", "request_id": ..., "command_preview": ..., "source": ...}`
- `handle_command_callback()` approve/deny 後刪除對應 schedule（避免誤通知）

### Phase 3: Notification（notifications.py）
- `send_expiry_warning_notification()`：發送 `⏰ 審批請求即將過期（60秒）：...`

### Phase 4: Tests
- mock `scheduler_service.create_schedule()` 驗證在 submit_for_approval 後被呼叫
- mock `scheduler_service.delete_schedule()` 驗證在 approve/deny 後被呼叫
- notification 格式 test

## Security Considerations
- Schedule payload 不含完整 command（只含 preview），避免 CloudWatch logs 洩露敏感命令
- IAM 最小權限：Lambda 只能操作自己建立的 schedules（用 request_id 作為 schedule name）
