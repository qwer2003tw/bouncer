# s37-001 Spec — fix: changeset count=0 不觸發人工確認

**Issue:** #124
**Sprint:** 37
**Type:** Bug Fix

---

## 問題描述

部署某些專案時，Steven 看到 Telegram 通知：「⚠️ Infrastructure Changes Detected — 🔧 變更數量: 0，需要人工確認」。  
變更數量為 0 卻要求人工確認，邏輯矛盾，造成不必要的摩擦。

---

## 根因分析

### 路徑 A：`analysis_error` 非空（exception path）

**File:** `deployer/notifier/app.py`，`handle_analyze()` 函數

```
line 319–325 (exception catch block):
    except Exception as exc:
        return {
            'is_code_only': False,   # ← fail-safe，但 change_count 也是 0
            'deploy_id': deploy_id,
            'project_id': project_id,
            'change_count': 0,       # ← 明確是 0
            'analysis_error': str(exc)[:256],
        }
```

當 `create_dry_run_changeset` 或 `analyze_changeset` 拋出 exception，回傳 `is_code_only=False` + `change_count=0`。  
SFN `CheckChangesetResult` Choice state（`deployer/template.yaml` line 708-715）：

```yaml
CheckChangesetResult:
  Type: Choice
  Choices:
    - Variable: $.analyze_result.is_code_only
      BooleanEquals: true
      Next: SamDeploy
  Default: WaitForInfraApproval  # ← is_code_only=False 走這裡
```

`WaitForInfraApproval` 呼叫 `handle_infra_approval_request`（line 366），  
取 `change_count = event.get('change_count', 0)` → 值為 0，  
發出「變更數量: 0，需要人工確認」的通知。

### 路徑 B：`missing stack_name / template_s3_url`

**File:** `deployer/notifier/app.py`，`handle_analyze()` line 281-289：

```python
if not stack_name or not template_s3_url:
    return {
        'is_code_only': False,
        'change_count': 0,
        'analysis_error': 'Missing stack_name or template_s3_url',
    }
```

同樣 `is_code_only=False` + `change_count=0` → 路由到 `WaitForInfraApproval`。

### 根因總結

`change_count == 0` 有兩種語義：
1. **changeset 分析成功，真的沒有變更**（no-op deploy）→ 應 auto-approve
2. **changeset 分析失敗（error / missing config）**→ 可能有變更但無法確認

目前的 `handle_infra_approval_request` 只顯示 `change_count`，不區分是哪種情況，用戶看到「0 個變更需要確認」感到困惑。

---

## 修法方向

`handle_infra_approval_request` 增加 `analysis_error` 感知：

1. **`analysis_error` 非空** + `change_count == 0`：  
   通知文字改為「⚠️ Changeset 分析失敗，無法確認變更範圍，請人工審查」  
   顯示 `analysis_error` 讓 Steven 知道原因
   
2. **`analysis_error` 為空** + `change_count == 0`：  
   理論上 `is_code_only_change()` 對空 changes 回傳 `True`（`changeset_analyzer.py` line 47: `if not result.resource_changes: return True`），  
   此 case 不應走到 `WaitForInfraApproval`；若走到，視為異常，通知文字說明。

3. **`change_count > 0`**：維持現有邏輯（顯示「變更數量: N」）

---

## Acceptance Scenarios

### Scenario 1: Changeset 分析 exception → 通知說明失敗原因
**Given** 部署 `bouncer` project  
**When** `create_dry_run_changeset` 拋出 `ClientError`（例如：`AccessDenied`）  
**Then**  
- SFN 路由到 `WaitForInfraApproval`（fail-safe 行為保留）  
- Telegram 通知顯示「⚠️ Changeset 分析失敗：{analysis_error}，無法確認變更範圍」  
- **不顯示** 「變更數量: 0」

### Scenario 2: Missing stack_name → 通知說明配置缺失
**Given** `bouncer-projects` DDB 表缺少 `template_s3_url` 欄位  
**When** `handle_analyze()` 執行  
**Then**  
- SFN 路由到 `WaitForInfraApproval`  
- Telegram 通知顯示「⚠️ 配置缺失：stack_name 或 template_s3_url 未設定，請人工審查」

### Scenario 3: 正常 infra 變更 → 顯示變更數量
**Given** 部署含 IAM Role 變更的 template  
**When** Changeset 分析成功，`change_count=3`  
**Then**  
- Telegram 通知顯示「🔧 變更數量: 3」（現有行為保留）

### Scenario 4: 正常 code-only → auto-approve（不進入 WaitForInfraApproval）
**Given** 部署只有 Lambda Function code 變更  
**When** `is_code_only_change()` 回傳 `True`  
**Then**  
- SFN 直接走 `SamDeploy`（現有行為保留，此 fix 不影響）

---

## Edge Cases

| Edge Case | 預期行為 |
|-----------|---------|
| `analysis_error` 很長（>200 字） | 截斷到 200 字顯示 |
| `analysis_error` = `'No updates are to be performed'` | 此 case 已在 handle_analyze 特殊處理（回傳 `is_code_only=True`），不走 WaitForInfraApproval |
| `change_count=0` 且 `analysis_error=None` | 說明 is_code_only 判斷異常，顯示「分析結果異常：0 個變更卻未 auto-approve」 |
| `analyze_result` key 不存在（SFN bug） | `handle_infra_approval_request` 使用 `.get()` 有預設值，安全 |
