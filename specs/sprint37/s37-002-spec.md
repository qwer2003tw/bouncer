# s37-002 Spec — feat: 審批通知顯示 CFN changeset 明細

**Issue:** #123
**Sprint:** 37
**Type:** Feature

---

## 問題描述

目前 infra approval 的 Telegram 通知只顯示「🔧 變更數量: N」，Steven 必須靠數字猜測要審查什麼。  
需要顯示每個 CFN 資源的 `LogicalResourceId`、`Action`、`Replacement`，讓 Steven 在不進 AWS Console 的情況下做出正確判斷。

---

## 根因分析（現狀）

### `handle_analyze()` 回傳值

**File:** `deployer/notifier/app.py`，line 309-316：

```python
return {
    'is_code_only': is_code_only,
    'deploy_id': deploy_id,
    'project_id': project_id,
    'change_count': len(analysis.resource_changes),
    'analysis_error': analysis.error,
}
```

`resource_changes`（CFN `Changes[]` raw dicts）**未被包含**在回傳值中，因此無法傳到 `handle_infra_approval_request`。

### `handle_infra_approval_request()` 通知

**File:** `deployer/notifier/app.py`，line 394-400：

```python
text = (
    f"⚠️ *Infrastructure Changes Detected*\n\n"
    f"📦 *專案：* {project_id}\n"
    f"🌿 *分支：* {branch}\n"
    f"🆔 *ID：* `{deploy_id}`\n\n"
    f"🔧 *變更數量：* {change_count}\n\n"   # ← 只有數字
    f"⚡ 偵測到 infra 變更，需要人工確認才能繼續部署。"
)
```

### CFN ResourceChange 結構

`AnalysisResult.resource_changes` 中每個元素格式（來自 `describe_change_set` API）：

```json
{
  "Type": "Resource",
  "ResourceChange": {
    "Action": "Modify",                    // Add / Modify / Delete
    "LogicalResourceId": "ApiFunction",
    "ResourceType": "AWS::Lambda::Function",
    "Replacement": "False",                // True / False / Conditional
    "Details": [...]
  }
}
```

### SFN 資料流

`deployer/template.yaml` line 725 將整個 `analyze_result` 傳入 `handle_infra_approval_request`：
```yaml
analyze_result.$: $.analyze_result
```
所以只要在 `handle_analyze()` 回傳的 dict 中加入 `resource_changes` 摘要，`handle_infra_approval_request` 即可讀到。

---

## 修法方向

1. **`handle_analyze()` 回傳加入 `resource_changes` 摘要**（截斷，避免 SFN payload 過大）  
   每個 change 只保留：`logical_id`、`action`、`resource_type`、`replacement`

2. **`handle_infra_approval_request()` 讀取摘要並格式化**  
   顯示每個資源一行：`{action_emoji} {LogicalResourceId} ({ResourceType}) {replacement_flag}`

3. **Telegram 訊息長度保護**：最多顯示 10 個資源，超過顯示「... 及 N 個其他變更」

---

## Acceptance Scenarios

### Scenario 1: 3 個 infra 資源變更 → 明細顯示
**Given** `handle_analyze()` 偵測到 3 個 CFN resource changes：
  - `ApiGateway`：Add，`AWS::ApiGateway::RestApi`
  - `LambdaRole`：Modify，`AWS::IAM::Role`，Replacement=True
  - `ApiFunction`：Modify，`AWS::Lambda::Function`，Replacement=False

**When** `handle_infra_approval_request()` 發送通知  
**Then** Telegram 通知包含：
```
📋 變更明細：
➕ ApiGateway (AWS::ApiGateway::RestApi)
✏️ LambdaRole (AWS::IAM::Role) ⚠️ 需替換
✏️ ApiFunction (AWS::Lambda::Function)
```

### Scenario 2: 超過 10 個資源 → 截斷顯示
**Given** 15 個 resource changes  
**When** 通知發送  
**Then** 顯示前 10 個，最後一行：`... 及 5 個其他變更`

### Scenario 3: `resource_changes` 為空列表（但 `is_code_only=False`）
**Given** `change_count=0`（analysis 失敗情境，見 s37-001）  
**When** 通知發送  
**Then** 不顯示「變更明細」區塊（由 s37-001 處理）

### Scenario 4: `resource_changes` key 不存在（舊版 SFN 紀錄）
**Given** 舊版 SFN execution 沒有 `resource_changes` 欄位  
**When** `handle_infra_approval_request()` 執行  
**Then** 回退到只顯示「🔧 變更數量: N」（向下相容）

---

## Edge Cases

| Edge Case | 預期行為 |
|-----------|---------|
| `Replacement = "Conditional"` | 顯示 `⚠️ 可能替換` |
| `Replacement = "True"` | 顯示 `⚠️ 需替換`（破壞性變更警示） |
| `LogicalResourceId` 超長（>40字） | 截斷加 `...` |
| `ResourceType` 未知格式 | 照原樣顯示 |
| SFN payload size limit（256KB） | 只存摘要欄位，不存完整 `Details[]` |
| `action` = Delete | 顯示 `🗑️ {id} 刪除` |
